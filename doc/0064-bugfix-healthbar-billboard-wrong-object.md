# 0064. 버그수정: 체력바가 카메라 반대 방향을 봄 (0063 후속 수정)

**날짜:** 2026-07-12

## 요청 내용
> [[0063-bugfix-duplicate-healthbar-billboard|0063]]에서 중복된 `HealthBarBillboard`를 정리했는데, 지금은 체력바가 카메라를 보는 게 아니라 X값 45로 고정되어 반대 방향을 보고 있다. 확인하고 고쳐달라.

## 원인 분석
0063에서는 `HealthBarBillboard`를 부모(`HealthBar`, 위치만 담당하는 래퍼)에 남기고 자식(`Canvas`, 실제로 렌더링되는 오브젝트)에서 제거했다. 그런데 이 순서가 반대였다.

`Canvas`는 원래 `HealthBar` 기준으로 로컬 X `-90도`가 authored돼 있는데(체력바 평면을 눕혀서 보여주기 위한 고정값), 부모만 매 프레임 월드 회전을 `(카메라 X, 0, 0)`으로 덮어쓰면 자식은 그 위에 **자기 고정 로컬 -90도를 그대로 더해서** 최종 월드 회전이 `(카메라X - 90, 0, 0)`이 되어버린다. 카메라 각도를 그대로 따라가는 게 아니라 거기서 90도 어긋난 각도로 보이니 반대 방향처럼 보인 것이다.

직접 좌표 계산으로 검증해보면(카메라가 X축으로 45도 아래를 보는 일반적인 탑다운 RTS 카메라 기준): 카메라의 forward 벡터는 `(0, -sin45°, cos45°)`이고, 이와 정확히 마주보려면(카메라 쪽을 향하는 노멀) 오브젝트의 회전이 **카메라와 동일한 X값**(추가 오프셋 없이)일 때 정확히 일치한다. 즉 애초에 `HealthBarBillboard`가 하던 계산(`transform.rotation = Quaternion.Euler(카메라의 X, 0, 0)`) 자체는 맞는데, 이걸 **실제로 렌더링되는 오브젝트(Canvas)가 아니라 그 부모(HealthBar)에 적용하고, Canvas에는 손대지 않아서** 목표와 다른 각도가 된 것이다.

## 수정 내용
`Assets/prefabs/UI/HealthBar.prefab`에서 `HealthBarBillboard`를 **`HealthBar`(부모)에서 제거하고 `Canvas`(자식, 실제 렌더링 오브젝트)로 옮겼다.**

`transform.rotation`(월드 회전)을 직접 대입하는 스크립트 특성상, 이제 `Canvas`의 월드 회전은 부모가 무슨 회전을 하든(유닛이 Y축으로 돌든 말든) 상관없이 항상 정확히 `(카메라의 X, 0, 0)`으로 고정된다 — 부모의 회전에 의존하지 않으므로 이번 같은 "부모+자식 오프셋이 겹쳐서 어긋나는" 문제 자체가 구조적으로 재발하지 않는다. `HealthBar`(부모)는 이제 순수하게 위치 오프셋(유닛 위 2미터)만 담당하는 래퍼로 남는다.
